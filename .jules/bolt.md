## 2024-03-20 - Scalar aggregate queries instead of .count()
**Learning:** In SQLAlchemy, `db.query(Model).filter(...).count()` generates inefficient subqueries (e.g., `SELECT count(*) FROM (SELECT ...)`). For simple single-table aggregate queries where we just want a count, using `db.scalar(select(func.count(Model.id)).where(...))` produces a direct, optimized aggregate count query (`SELECT count(Model.id) FROM Model WHERE ...`), significantly reducing database CPU load and execution time. However, this optimization CANNOT be safely applied to generic/arbitrary queries (like inside a pagination utility) because using `.with_entities(func.count())` or replacing `.count()` breaks queries containing `GROUP BY` or `DISTINCT` by improperly overriding entities.
**Action:** When performing simple aggregate count queries in business logic services, prefer `db.scalar(select(func.count(Model.id)).where(...))` instead of `db.query(Model).filter(...).count()`. For generic query handling (like pagination wrappers), stick with `.order_by(None).count()` to safely remove sorting overhead while allowing SQLAlchemy to properly wrap and manage subqueries for complex groupings.
