## 2025-01-31 - Missing Rate Limiting on Some Authentication and MFA Endpoints
**Vulnerability:** Found that while most critical MFA endpoints (`verify_totp_setup`, `complete_mfa_challenge`, `verify_mfa_code`, `initiate_duo_auth`, `verify_duo_callback`) had proper rate limiting applied via `@limiter.limit(f"{settings.RATE_LIMIT_AUTH}/minute")`, some secondary sensitive actions (`setup_totp`, `regenerate_recovery_codes`, `disable_mfa`) did not have it. This inconsistent application of rate limiting leaves certain pathways open to abuse (e.g. DoS by rapidly calling the setup endpoint and exhausting the database or server resources, or brute-forcing token generation).
**Learning:** Even within the same router file that implements an architecture-level defense, the omission of decorators on secondary actions often goes unnoticed. The `slowapi` library requires the `request: Request` argument in the function signature for `@limiter.limit` to extract the client IP; developers sometimes skip rate limiting to avoid boilerplate or forget to add the required `Request` dependency on endpoints that don't otherwise use it.
**Prevention:** Establish a strict policy and automated checks (e.g., custom linter or AST checker) to ensure that all `POST`/`PUT`/`DELETE` endpoints inside authentication, MFA, and profile management routers explicitly define rate limiting logic.
